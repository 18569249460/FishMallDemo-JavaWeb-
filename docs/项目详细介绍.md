# 鲜渔小店项目详细介绍

## 一、项目概述

“鲜渔小店”是一个面向水产品线上销售场景的商城管理系统。项目以用户端商城和管理端后台为核心，覆盖商品浏览、分类筛选、购物车、下单、订单状态流转、个人中心、地址管理、图片上传，以及后台商品、分类、订单、用户维护等常见业务。

项目采用前后端分离开发方式：

- 前端使用 Vue 3、Vue Router、Pinia、Element Plus、Axios、Vite。
- 后端使用 SSM 架构，即 Spring、SpringMVC、MyBatis。
- 数据库使用 MySQL，连接池使用 Druid。
- 登录认证使用 JWT，密码按当前项目要求明文保存。
- 后端以 WAR 包形式部署到 Tomcat，前端开发阶段通过 Vite 代理访问后端接口。

当前项目已经不是只有页面壳子的初始版本，而是具备完整业务流程的水产品商城雏形：普通用户可以注册登录、浏览商品、加入购物车、下单、查看订单、维护地址和上传头像；管理员可以进入后台维护商品、分类、订单、用户状态，上传商品图片，并可在后台新增管理员账号。

## 二、适用角色

项目主要面向两类角色。

### 1. 普通用户

普通用户通过前台页面完成购物流程，包括：

- 注册、登录。
- 浏览首页推荐商品。
- 按分类和关键词筛选商品。
- 查看商品详情。
- 加入购物车。
- 管理购物车商品数量。
- 批量删除购物车商品。
- 清理失效商品。
- 选择常用地址并提交订单。
- 查看订单列表和订单详情。
- 取消待发货订单。
- 确认收货。
- 编辑个人资料和上传头像。
- 维护常用收货地址。

### 2. 管理员

管理员登录后进入后台管理系统，主要负责平台运营维护，包括：

- 查看后台首页数据概览。
- 查看用户总数、商品总数、订单总数、待处理订单数。
- 查看近七天销量折线图。
- 查看当天待发货订单占比。
- 管理商品分类。
- 管理商品信息、库存、状态和图片。
- 上传商品图片到商品图片目录。
- 管理订单状态和物流单号。
- 管理普通用户启用/禁用状态。
- 在账户管理中新增管理员账号。

## 四、整体架构

项目整体分为三层：

```text
浏览器前端 Vue 3
        |
        | Axios /api 请求
        v
SpringMVC Controller
        |
        | 调用 Service
        v
Service 业务层
        |
        | 调用 Mapper
        v
MyBatis Mapper + XML SQL
        |
        v
MySQL 数据库
```

前端开发时访问：

```text
http://localhost:5173
```

Vite 会把 `/api` 请求代理到后端：

```text
http://localhost:8080/fishMall/api
```

后端部署上下文推荐为：

```text
/fishMall
```

## 六、核心功能介绍

### 1. 认证模块

认证模块包括：

- 用户注册。
- 用户登录。
- JWT 返回。
- 当前用户信息查询。

后端关键类：

- `AuthController`
- `AuthServiceImpl`
- `UserController`
- `UploadController`
- `AuthInterceptor`
- `JwtUtil`
- `PasswordUtil`

前端关键文件：

- `LoginView.vue`
- `stores/user.js`
- `router/index.js`
- `api/index.js`

登录成功后，前端会把 Token 和用户信息保存到 `localStorage`。后续请求通过 Axios 请求拦截器自动携带：

```text
Authorization: Bearer <token>
```

管理员账号登录后默认跳转后台首页：

```text
/admin
```

普通用户登录后进入前台页面。

管理员账号不通过前台注册生成。第一位管理员可通过 `docs/测试数据.sql` 初始化，后续管理员由已登录管理员在后台“账户管理”页面新增。

图片上传由 `UploadController` 统一处理：

- 头像上传接口为 `/api/uploads/avatar`，普通登录用户可使用。
- 商品图片上传接口为 `/api/uploads/product`，后端会额外校验管理员角色。
- 头像保存到 `/uploads/images/touxiang`，商品图片保存到 `/uploads/images/shangpin`，数据库只保存返回的图片访问地址。
- 头像文件名使用 `avatar_` 前缀，商品图片文件名使用 `product_` 前缀，便于后续排查和清理。

## 七、数据库设计

当前数据库名：

```text
fish_shop_db
```

主要数据表：

| 表名 | 说明 |
| :--- | :--- |
| `user` | 用户表，保存普通用户和管理员 |
| `category` | 商品分类表 |
| `product` | 商品表 |
| `user_address` | 常用收货地址表 |
| `cart` | 购物车表 |
| `orders` | 订单主表 |
| `order_item` | 订单商品快照表 |

### 1. user 用户表

核心字段：

- `id`：用户 ID。
- `avatar`：头像地址。
- `nickname`：昵称。
- `phone`：手机号。
- `password`：明文密码。
- `role`：角色，`0` 普通用户，`1` 管理员。
- `status`：状态，`1` 启用，`0` 禁用。

### 2. category 分类表

核心字段：

- `id`：分类 ID。
- `name`：分类名称。
- `sort_order`：排序值。
- `status`：启用状态。

### 3. product 商品表

核心字段：

- `id`：商品 ID。
- `category_id`：所属分类。
- `name`：商品名称。
- `price`：价格。
- `stock`：库存。
- `specification`：规格。
- `image_url`：商品图片。
- `description`：描述。
- `origin`：产地。
- `shelf_life`：保质期。
- `storage_method`：储存方式。
- `delivery_info`：配送说明。
- `sales`：销量。
- `status`：上下架状态。

### 4. cart 购物车表

核心字段：

- `user_id`：用户 ID。
- `product_id`：商品 ID。
- `quantity`：数量。

`user_id + product_id` 有唯一约束，同一用户对同一商品只保留一条购物车记录。

### 5. orders 订单表

核心字段：

- `order_no`：订单号。
- `user_id`：下单用户。
- `receiver_name`：收货人。
- `receiver_phone`：收货电话。
- `address_detail`：收货地址。
- `tracking_no`：物流单号。
- `total_amount`：订单总金额。
- `status`：订单状态。

### 6. order_item 订单商品快照表

核心字段：

- `order_id`：订单 ID。
- `product_id`：商品 ID。
- `product_name`：下单时商品名称。
- `price`：下单时价格。
- `quantity`：购买数量。
- `subtotal`：小计金额。

该表保存商品快照，即使后续商品被修改或删除，历史订单仍能显示下单时的商品信息。

## 九、权限和安全设计

### 1. 登录认证

用户登录成功后，后端返回 JWT。前端保存 JWT，并在后续请求中放入请求头。

后端 `AuthInterceptor` 会完成：

- 判断是否携带 Token。
- 校验 Token 是否有效。
- 查询当前用户是否存在。
- 判断用户是否被禁用。
- 对 `/api/admin/**` 接口校验管理员角色。

商品图片上传接口虽然不是 `/api/admin/**` 路径，但后端会在 `UploadController` 中读取当前用户角色并再次校验管理员权限，避免只依赖前端页面隐藏按钮。

### 2. 密码存储说明

当前项目按使用要求采用明文密码存储。注册和后台新增管理员时，`password` 字段直接保存用户输入的密码；登录时用用户输入的密码与数据库中的 `password` 做精确匹配。

这种方式便于课程设计阶段查看和维护数据，但不适合真实生产环境。后续如果要上线，应改回不可逆密码哈希方案。

### 3. 参数校验

后端 DTO 使用：

- `@NotBlank`
- `@NotNull`
- `@Size`
- `@Pattern`
- `@Min`
- `@DecimalMin`

前端页面也做基础中文校验提示，例如手机号格式统一提示：

```text
请输入正确的手机格式
```

### 4. 事务设计

下单、取消订单等涉及多表更新的操作使用事务保护，确保库存、销量、订单、订单商品快照保持一致。

典型事务场景：

- 创建订单：生成订单、写订单商品、扣库存、加销量、删除购物车项。
- 取消订单：订单状态变更、库存回滚、销量回滚。

## 十五、当前限制与后续优化方向

当前项目已经具备完整演示流程，但如果要进一步接近生产系统，还可以继续优化：

1. 后台列表分页目前多为前端本地分页，后续可以改为后端分页查询。
2. 头像和商品图片目前保存到 Tomcat Web 应用的 `/uploads/images/touxiang`、`/uploads/images/shangpin` 目录，生产环境建议改为独立文件目录或对象存储，避免重新部署覆盖上传文件。
3. 商品图片目前支持上传和填写外部 URL，后续可以增加图片删除、旧图清理和图片压缩能力。
4. 订单物流目前只有物流单号，后续可以增加物流公司、物流轨迹。
5. 缺少支付模块，后续可以增加模拟支付或第三方支付接口。
6. 缺少退款和售后流程，后续可以增加售后订单状态。
7. 缺少测试用例，后续可以补充 Service 层单元测试和接口集成测试。
8. 前端构建时提示包体积较大，后续可以做路由懒加载和手动分包。
9. 后台权限目前只有管理员和普通用户两级，后续可以细分运营、客服、仓库等角色。
10. 数据统计可以继续扩展为销售额趋势、热销分类、库存预警等图表。
